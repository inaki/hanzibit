# Beginner Onboarding Plan

This document turns the first-user journey into a concrete beginner onboarding plan for HanziBit.

The reference learner is:

- starting Chinese from zero
- not joining a class or teacher flow initially
- using the product on web first
- trying to understand what to do in the first 5 minutes

## Goal

The first-run experience should answer three questions immediately:

1. What is HanziBit for?
2. What should I do first?
3. Why should I come back tomorrow?

## Onboarding principles

### 1. Do not front-load complexity

Avoid showing, explaining, or emphasizing:

- teacher workflows
- admin/platform language
- advanced reporting ideas
- too many navigation destinations
- heavy HSK framing before the learner has context

### 2. Start with one small success

The first session should guide the learner to:

- open one study item
- write one tiny guided response
- complete one tiny review

### 3. Explain through motion

Instead of a long intro, the UI should teach through:

- one short welcome message
- one highlighted first action
- contextual tips inside the flow

## First-run experience

### Step 1: Dashboard welcome state

When a new learner lands on `/notebook/dashboard` with little or no data:

Show:

- a short welcome message
- one-sentence explanation of the daily loop
- one primary CTA: `Start with one word`

Avoid:

- dense stat cards with zeros everywhere
- unexplained HSK jargon
- too many parallel actions

Recommended copy direction:

- `Start with one word, write one short response, and review a few cards.`

### Step 2: Beginner study entry

The first CTA should open a beginner-safe Study Guide experience:

- one HSK 1 word
- compact meaning
- pinyin
- one tiny example
- one obvious next action: `Use this in journal`

The learner should not need to search before getting value.

### Step 3: Guided journal entry

The journal should open with:

- the word already seeded
- a tiny prompt
- optional annotation help

The learner should feel:

- supported
- not graded
- not blocked by syntax

### Step 4: Tiny review

After saving the first entry, guide the learner into:

- a very short review action
- ideally 1 to 3 items

This should reinforce the feeling that the app remembers what they studied.

### Step 5: Return-path cue

At the end of the first session, make tomorrow obvious:

- `Come back tomorrow and continue Today’s Practice.`
- or `Tomorrow we’ll review this word and add one more.`

## UI changes implied by this plan

### Dashboard

Needs:

- a true beginner welcome state
- a more instructional empty state
- fewer passive zero-value stats competing with the first action

### Study Guide

Needs:

- a beginner-first entry path
- reduced panel overload for the first viewed word
- maybe a compact beginner mode on first use

### Daily Journal

Needs:

- optional annotation help that feels supportive, not mandatory
- a clearer beginner explanation of why writing matters

### Flashcards

Needs:

- a tiny first review feel
- stronger connection to what was just studied

## Suggested implementation order

### 1. Beginner dashboard first-run state

Highest leverage. This determines whether a new learner understands the product at all.

### 2. Beginner-first Study Guide launch

Make the first study interaction compact and obvious.

### 3. First guided journal micro-flow

Ensure the first writing task is small and not intimidating.

### 4. First review micro-flow

Keep the first review tiny and clearly connected.

### 5. Return cue / day-two cue

Reinforce habit formation.

## What should probably wait

Not necessary for the first beginner pilot:

- teacher-connected flows
- deep reporting concepts
- advanced notebook tooling
- broad curriculum settings
- rich personalization before the learner has even used the loop once

## Success criteria

A new learner should be able to:

- understand the product within 30 to 60 seconds
- complete one meaningful first session without confusion
- know the next step without searching around the app
- feel the app is simple enough to return to tomorrow
